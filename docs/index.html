Instalación

Dos pasos: instalar el comando global una vez, luego inicializar cada proyecto.

1

Instalar el comando global /opsx-init

Hacelo una sola vez en tu máquina. Instala el comando en Claude Code de forma global para que esté disponible en cualquier proyecto.

# Una sola vez — instala /opsx-init globalmente
mkdir -p ~/.claude/commands
curl -fsSL https://raw.githubusercontent.com/devu2pi/utopia-stdd-registry/main/commands/opsx-init.md \
  -o ~/.claude/commands/opsx-init.md
2

Inicializar un proyecto

Abrí Claude Code en la carpeta del proyecto (nuevo o existente) y ejecutá:

# En Claude Code, parado en la raíz del proyecto
/opsx-init

Claude detecta si el proyecto es nuevo o existente, pregunta el archetype si hace falta, y configura todo automáticamente.

3

Completar el contexto del proyecto

El único paso manual: completar openspec/project.md con la descripción real del sistema, módulos y decisiones técnicas. Claude lo lee al inicio de cada sesión.

# Verificar estado inicial
/opsx-status

Workflow

Cada cambio de código sigue este ciclo — sin excepciones.

"Acordá antes de construir. El spec lidera, el código sigue." — Filosofía OpenSpec
1
/opsx-propose <nombre>

Proponer

Creá una propuesta con qué se cambia, por qué, qué incluye y qué no. Claude valida que openspec/project.md esté completo antes de continuar.

✓
⚠ Gate

La proposal debe ser aprobada explícitamente antes de avanzar. Claude la revisa y marca como APPROVED.

2
/opsx-apply <nombre>

Aplicar

Genera spec.md (escenarios de comportamiento), design.md (decisiones técnicas) y tasks.md (checklist de implementación).

3
/opsx-tdd-plan <nombre>

Planificar tests

Define los tests requeridos derivados de los escenarios de la spec. Cada tarea de implementación debe tener al menos un TEST-XX asociado.

⚠ Gate — no implementar sin tdd:plan completo
4
implementar

Implementar

Código y tests en paralelo, siguiendo el tasks.md generado. Sin sorpresas: todo lo que hay que hacer ya está listado.

5
/opsx-tdd-verify <nombre>

Verificar

Ejecuta los comandos de test del stack, verifica criterios de aceptación, chequea alineación con el registry. Produce VERIFIED ✓ o BLOCKED con motivo.

⚠ Gate — no archivar sin VERIFIED ✓
6
/opsx-archive <nombre>

Archivar

Mergea la spec al source of truth en openspec/specs/, actualiza project.md con las decisiones tomadas y sugiere el commit de cierre.

Estados de un cambio

DRAFT
Propuesta creada, pendiente de completar
APPROVED
Proposal aprobada, lista para /opsx-apply
APPLIED
Artefactos generados, lista para tdd:plan
TDD_PLANNED
Tests definidos, en implementación
BLOCKED
Verificación falló, requiere correcciones
TDD_VERIFIED
Todos los tests pasan, listo para archive
ARCHIVED
Cambio cerrado, spec mergeada

Skills

Archivos Markdown inyectados como contexto en cada sesión de Claude Code. Se consumen desde GitHub automáticamente.

openspec/v1

OpenSpec Base

Metodología SDD core. Workflow, estructura de cambios y reglas de gobernanza. Wrapper sobre Fission-AI/OpenSpec.

tdd-extension/v1

TDD Extension

Agrega fases tdd:plan y tdd:verify al workflow. Define niveles de aprobación (1=auto, 2=par, 3=arquitectura).

architecture/v1

Architecture

Patrones aprobados: capas, data fetching, estado, auth, base de datos. ADRs centrales del equipo.

stack/v1

Stack

Tecnologías aprobadas con versiones, servicios externos y dependencias a evitar.

code-review/v1

Code Review

Checklist y estándares de revisión de código aplicados en la fase tdd:verify.

# .agent/config.yaml — cómo un proyecto declara sus skills
registry:
  source: github
  repo: devu2pi/utopia-stdd-registry
  ref: main

skills:
  openspec:      "v1"
  tdd-extension: "v1"
  architecture:  "v1"
  stack:         "latest"
  code-review:   "v1"

Contribuir al registry

Cualquier cambio al registry impacta todos los proyectos que consuman skills desde aquí.

1

Crear branch

git checkout -b skill/nombre-version
2

Editar o crear el skill

# Nuevo skill
skills/nombre/v1.md

# Nueva versión de skill existente (breaking change)
skills/nombre/v2.md
3

Actualizar registry.json y abrir PR

Describir el cambio y listar los proyectos impactados. Merge → propagación automática a proyectos con latest.
