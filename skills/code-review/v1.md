# Code Review — Estándares y Checklist

## Principios de revisión

- El objetivo es mejorar el código, no demostrar que el revisor sabe más
- Los comentarios van al código, no a la persona
- Si algo no está claro, preguntar antes de asumir
- Un PR debe ser revisable en < 30 minutos; si no, es muy grande

---

## Checklist de revisión

### Corrección
- [ ] El código hace lo que el spec/task describe
- [ ] Los edge cases identificados en tasks.md están manejados
- [ ] No hay lógica de negocio hardcodeada que debería ser configurable
- [ ] Los errores se manejan explícitamente (no se silencian)

### Calidad
- [ ] Nombres de variables/funciones son descriptivos y en inglés
- [ ] No hay código comentado sin motivo (`// TODO` documentados están bien)
- [ ] No hay duplicación obvia (DRY aplicado razonablemente)
- [ ] Las funciones hacen una sola cosa
- [ ] Los archivos tienen una responsabilidad clara

### TypeScript / Python
- [ ] No hay `any` sin justificación comentada (TS)
- [ ] Los types/interfaces están bien definidos, no son `object` genérico
- [ ] Type hints completos en funciones Python
- [ ] Pydantic schemas usados para validación de entrada (Python)

### Tests
- [ ] Los tests describen comportamiento, no implementación
- [ ] Tests cubren el happy path y al menos un error path
- [ ] No hay lógica condicional en los tests
- [ ] Los mocks son mínimos y necesarios

### Seguridad
- [ ] No hay secrets o credenciales en el código
- [ ] Inputs del usuario son validados antes de usarse
- [ ] RLS activado si hay acceso directo a Supabase desde cliente
- [ ] No hay queries SQL construidas con string concatenation

### Performance
- [ ] No hay N+1 queries obvias
- [ ] Imágenes tienen tamaño/formato apropiado
- [ ] No hay re-renders innecesarios en componentes React

---

## Niveles de comentario en PR

Usar prefijos para dar contexto:

```
nit:      → detalle menor, no bloquea. "nit: podría ser const en lugar de let"
question: → duda genuina, no bloquea. "question: por qué se eligió X acá?"
suggest:  → sugerencia, no bloquea. "suggest: esto podría extraerse a un hook"
issue:    → problema real, bloquea. "issue: este caso no está cubierto"
blocker:  → problema crítico, bloquea. "blocker: esto rompe el contrato de la API"
```

---

## Tamaño de PR recomendado

```
Ideal:    < 400 líneas cambiadas
Aceptable: 400–800 líneas (con descripción clara)
Requiere división: > 800 líneas
```

Si un cambio necesariamente supera 800 líneas, dividir en:
1. PR de infraestructura/setup (sin lógica de negocio)
2. PR de implementación
