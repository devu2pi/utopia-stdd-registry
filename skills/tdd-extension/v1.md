# TDD Extension — Verificación y Aprobación

## Filosofía

> El spec define el contrato. Los tests verifican que el contrato se cumple.

Esta extensión agrega puntos de control basados en tests al workflow de OpenSpec.
El objetivo no es generar tests automáticamente sino asegurar que **ningún cambio
se archive sin haber pasado por verificación explícita**.

---

## Workflow extendido

```
/stdd-propose    → (igual que OpenSpec base)
/stdd-apply      → (igual que OpenSpec base) + genera sección de tests en tasks.md
/stdd-plan   → define qué tests se necesitan ANTES de implementar
/stdd-verify → ejecuta verificación y genera reporte
/stdd-archive    → solo disponible si tdd:verify pasó ✓
```

---

## Fase: tdd:plan

Se ejecuta después de `/stdd-apply`, antes de implementar.

**Qué produce:** una sección `## Tests Requeridos` en `tasks.md`:

```markdown
## Tests Requeridos

### Unitarios
- [ ] TEST-01: [descripción del comportamiento a verificar]
      Input: [qué entra]
      Expected: [qué debe devolver]
      Archivo: tests/unit/<módulo>.test.ts

### Integración  
- [ ] TEST-02: [flujo end-to-end a verificar]
      Precondición: [estado inicial]
      Acción: [qué se ejecuta]
      Expected: [resultado esperado]
      Archivo: tests/integration/<feature>.test.ts

### Criterios de aceptación
- [ ] ACC-01: [criterio en lenguaje de negocio]
```

**Regla:** cada tarea de implementación en `tasks.md` debe tener al menos un TEST- asociado.

---

## Fase: tdd:verify

Se ejecuta después de implementar, antes de `/stdd-archive`.

**Checklist de verificación:**

```markdown
## Reporte de Verificación — <nombre-del-cambio>

### Tests
- [ ] Todos los TEST- de tasks.md tienen implementación
- [ ] Tests unitarios pasan: `npm run test:unit` ✓/✗
- [ ] Tests de integración pasan: `npm run test:integration` ✓/✗
- [ ] Coverage no bajó del umbral definido en el proyecto

### Criterios de aceptación
- [ ] Todos los ACC- fueron verificados manualmente o por test

### Code review
- [ ] Sigue los patrones de `architecture/v1.md`
- [ ] Usa stack aprobado en `stack/v1.md`
- [ ] Pasa checklist de `code-review/v1.md`

### Regresión
- [ ] No se rompieron tests existentes
- [ ] Funcionalidad previa relacionada fue revisada

### Aprobación
- [ ] Auto-aprobación (cambio menor, sin impacto en otros módulos)
- [ ] Requiere revisión de par (cambio en módulo crítico)
- [ ] Requiere aprobación de arquitectura (cambio de patrón o stack)
```

**Resultado:** `VERIFIED ✓` o `BLOCKED — motivo`.

---

## Niveles de aprobación

```
NIVEL 1 — Auto-aprobación
  Condición: cambio acotado a un módulo, sin nuevas dependencias
  Quién aprueba: el mismo desarrollador
  
NIVEL 2 — Revisión de par
  Condición: cambio en módulo compartido, nueva integración, 
             modificación de API existente
  Quién aprueba: otro desarrollador del equipo
  
NIVEL 3 — Aprobación de arquitectura  
  Condición: cambio de stack, nuevo patrón, decisión que impacta
             múltiples proyectos
  Quién aprueba: equipo de arquitectura
```

## Integración con tasks.md

El formato extendido de `tasks.md` con TDD:

```markdown
# Tasks — <nombre-del-cambio>

## Implementación

- [ ] 1.1 Crear modelo de datos
      Test asociado: TEST-01
- [ ] 1.2 Implementar endpoint POST /api/recurso
      Test asociado: TEST-02, TEST-03
- [ ] 1.3 Implementar lógica de negocio
      Test asociado: TEST-04

## Tests (completar ANTES o EN PARALELO a implementación)

- [ ] TEST-01: modelo persiste correctamente
- [ ] TEST-02: endpoint devuelve 201 con body correcto
- [ ] TEST-03: endpoint devuelve 400 con input inválido
- [ ] TEST-04: lógica de negocio maneja edge cases

## Criterios de aceptación

- [ ] ACC-01: usuario puede crear un recurso desde la UI
- [ ] ACC-02: errores de validación se muestran correctamente

## Verificación (completar antes de /stdd-archive)

- [ ] tdd:verify ejecutado: ✓/✗
- [ ] Nivel de aprobación requerido: [1/2/3]
- [ ] Aprobado por: [nombre o "auto"]
```

## Comandos de referencia por stack

```bash
# Next.js / TypeScript
npm run test:unit          # Vitest o Jest
npm run test:integration   # Playwright o Cypress
npm run test:coverage      # umbral recomendado: 80%

# FastAPI / Python
pytest tests/unit/
pytest tests/integration/
pytest --cov=app --cov-report=term-missing

# General
npm run typecheck          # antes de cualquier archive
```
