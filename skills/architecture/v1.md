# Architecture — Patrones y Principios Centrales

## Principios base

1. **Separation of concerns** — cada módulo tiene una responsabilidad única y clara
2. **Dependency inversion** — depender de abstracciones, no de implementaciones concretas
3. **Fail fast** — validar en el borde del sistema, no en el interior
4. **Explicito sobre implícito** — el comportamiento debe ser legible sin conocer el contexto
5. **Brownfield-safe** — los cambios deben ser incrementales y reversibles

---

## Estructura de capas (aplicaciones web)

```
src/
├── app/              ← routing, layouts, páginas (Next.js App Router)
├── components/       ← UI components reutilizables
│   ├── ui/           ← componentes genéricos (Button, Input, Modal)
│   └── features/     ← componentes de dominio (UserCard, InvoiceTable)
├── lib/              ← lógica de negocio, sin dependencia de UI
│   ├── actions/      ← server actions o mutations
│   ├── queries/      ← read operations / data fetching
│   └── utils/        ← helpers puros (sin side effects)
├── hooks/            ← React hooks personalizados
├── types/            ← TypeScript types e interfaces
└── config/           ← configuración de la app
```

## Patrones aprobados

### Data fetching
- Server Components para fetch inicial (no useEffect para datos)
- React Query / SWR para estado del cliente con revalidación
- Server Actions para mutations

### Estado
- Zustand para estado global del cliente (evitar Context para estado frecuente)
- URL state (nuqs) para filtros y paginación
- Server state siempre source of truth

### Autenticación
- Supabase Auth como proveedor estándar
- Middleware de Next.js para protección de rutas
- No reinventar manejo de sesiones

### Base de datos
- Supabase PostgreSQL como estándar
- Row Level Security (RLS) activado en todas las tablas
- Migrations versionadas (nunca editar schema directamente en producción)
- Naming: snake_case en BD, camelCase en TypeScript

### APIs internas
- REST para operaciones CRUD simples
- Server Actions preferido sobre API routes para mutaciones desde UI
- API routes para integraciones externas o webhooks

### Error handling
- Errors deben surfacearse al borde (no silenciarse internamente)
- Usar Result pattern o typed errors en lógica de negocio crítica
- Siempre loguear con contexto suficiente para debuggear

## Decisiones de diseño (ADRs)

### ADR-001: Next.js App Router como estándar
- **Decisión:** usar App Router (no Pages Router) en proyectos nuevos
- **Motivo:** mejor soporte para Server Components, streaming, layouts anidados
- **Consecuencia:** no mezclar App Router y Pages Router en el mismo proyecto

### ADR-002: Supabase sobre soluciones custom de auth/db
- **Decisión:** Supabase como BaaS estándar para auth + PostgreSQL
- **Motivo:** reduce tiempo de setup, RLS nativo, buena integración con Next.js
- **Consecuencia:** proyectos con requerimientos de auth muy custom necesitan justificación

### ADR-003: TypeScript strict mode
- **Decisión:** `"strict": true` en todos los proyectos TypeScript
- **Motivo:** errores de tipo en compilación > errores en runtime
- **Consecuencia:** no usar `any` sin justificación explícita en comentario

## Patrones a evitar

```
❌ useEffect para data fetching inicial
❌ Lógica de negocio en componentes UI
❌ any en TypeScript sin comentario justificado
❌ Mutations directas a BD desde el cliente (bypassear RLS)
❌ Variables de entorno hardcodeadas en código
❌ Commits directos a main sin PR
```
