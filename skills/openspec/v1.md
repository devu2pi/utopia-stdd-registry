# OpenSpec — Skill Base

## Filosofía

> Acuerda antes de construir. El spec lidera, el código sigue.

OpenSpec agrega una capa de especificación estructurada antes de escribir código.
Evita el "vibe coding" — prompts vagos con resultados impredecibles.

---

## Workflow central

```
/stdd-explore   → analizar codebase, discutir approaches (sin artefactos)
/stdd-propose   → crear carpeta de cambio con proposal.md
/stdd-apply     → generar spec.md, design.md, tasks.md
/stdd-verify    → validar implementación contra spec (ver tdd-extension)
/stdd-archive   → mergear spec delta al source of truth
```

---

## Estructura de un cambio

```
openspec/changes/<nombre-del-cambio>/
├── proposal.md      ← qué se propone y por qué
├── specs/           ← requisitos y escenarios
│   └── <feature>.md
├── design.md        ← approach técnico y decisiones
└── tasks.md         ← checklist de implementación
```

## Reglas de gobernanza (capa empresarial)

Estas reglas se aplican sobre el workflow base de OpenSpec:

### Antes de proponer
- Revisar si existe un cambio activo relacionado (`openspec list`)
- Consultar `openspec/project.md` para entender el contexto del proyecto
- Alinear con los patrones definidos en la skill `architecture`

### Al escribir proposal.md
- Incluir sección **Impacto**: qué módulos/servicios se ven afectados
- Incluir sección **Riesgos**: qué puede salir mal
- Referenciar el arquetipo del proyecto si aplica

### Al escribir design.md
- Las decisiones técnicas deben alinearse con `stack/v1.md`
- Decisiones de arquitectura que se desvíen del estándar → requieren justificación explícita
- Si el cambio agrega una nueva dependencia → documentarla con motivo

### Al escribir tasks.md
- Cada tarea debe ser verificable (tiene un criterio de "done" claro)
- Tareas de implementación deben incluir tarea de test correspondiente
- Máximo granularidad: una tarea = ≤ 4 horas de trabajo

### Antes de archivar
- La skill `tdd-extension` define los pasos de verificación requeridos
- No archivar si hay tasks sin completar `[ ]`
- No archivar si los tests de verificación no pasaron

## Archivo openspec/project.md

Cada proyecto debe tener este archivo poblado antes de empezar a trabajar:

```markdown
# Contexto del Proyecto

## Descripción
[Qué hace el proyecto, para quién, qué problema resuelve]

## Stack
[Tecnologías, versiones, servicios externos]

## Arquitectura
[Estructura de carpetas, capas, patrones principales]

## Convenciones
[Naming, formatting, testing approach]

## Decisiones importantes
[ADRs relevantes, por qué se eligió X sobre Y]
```
